Launcher加载流程

发表于 2016-01-03 | 阅读次数:

概述

Launcher在AndroidManifest.xml中加入了category.home属性，因此开机的时候最先找到这个应用程序加载

和普通的应用程序一样，先从onCreate()开始分析

准备工作

这里先获取到LauncherAppState的一个实例(单例模式)，在该实例的构造函数中对许多变量进行了初始化同时注册了广播监听

这里贴的是CM的Launcher而并非原生,CM基于原生做了一些定制,例如这里的未读广播,系统中的其他应用可以通过向Launcher发送该广播,提醒Launcher更新桌面的应用图标,显示未读数量.
继续回到Launcher的onCreate()方法

这里获取到一个InvariantDeviceProfile对象,可以把该对象理解成手机的具体配置信息,在InvariantDeviceProfile类中保存了市面上主流的一些手机屏幕的大小,以及默认的布局格式,例如几行几列,应用图标的大小,字体大小等等,Launcher会从中选择一块最接近当前手机的屏幕大小,加载默认的配置文件,具体之后再分析.

接着获取一些在LauncherAppState中初始化过的重要对象后:

Launcher也是一个Activity,也需要加载布局,在layout-land和layout-port中都存在launcher.xml分别对应横屏和竖屏,在使用android平板或者屏幕很大的时候会用到横屏,这里我们就分析平时常用的竖屏的情况,Launcher中自定义了很多View,代码量都很多.在setupViews()中就是对布局中的控件各种findViewById,注册点击事件,还有拖拽事件(DragController);mDeviceProfile.layout(this)中根据之前InvariantDeviceProfile找到的最适合的一块手机屏幕确定布局中的控件的位置

  • Layout the search bar space(确定搜索框)
  • Layout the workspace(确定桌面,中间可以左右滑动的那块)
  • Layout the hotseat(确定热键区域)
  • Layout the page indicators(确定桌面指示的那个小点)
  • Layout the Overview Mode(确定长按进入的预览模式)

接下来就是采用异步的方式加载和绑定桌面和菜单里的应用、小部件、文件夹等,Launcher的加载流程也就是从这里开始

数据的加载和绑定

之前说了Launcher的数据加载和绑定的流程是从LauncherModel的startLoader开始的,因此这里重点开始分析这一流程

这里的mLoaderTask是LauncherModel的内部类,实现了Runnable,第一次开机的时候条件判断执行上面的else中的代码,走到LoaderTask中执行run()方法:

这里分为两个步骤:

  • loadAndBindWorkspace(); //加载和绑定桌面
  • loadAndBindAllApps();//加载和绑定菜单

先分析桌面上数据的加载和绑定

loadAndBindWorkspace()

桌面上数据分为加载和绑定:

  • loadWorkspace() //加载和绑定桌面
  • bindWorkspace(-1)//加载和绑定菜单

loadWorkspace()

第一次开机时launcher的数据库为空，因此要加载默认的配置文件：

查看loadDefaultFavoritesIfNecessary()中是如何加载默认配置:

首先获取布局加载器loader对默认的布局进行加载，调用DatabaseHelper的loadFavorites方法加载默认的布局文件(default_workspace.xml)。回到loadworkspace()中，这个方法的代码很多，但是基本思路很清晰，在default_workspace.xml中我们自定义了一些要在桌面上显示的ShortcutInfo，这里的ShortcutInfo包括快捷方式、小部件、文件夹等，它们所在的区域可能在中间(CONTAINER_DESKTOP),也可能在热键区(CONTAINER_HOTSEAT)，这里的工作就是遍历所有之前默认配置文件中的内容，保存在对应的数据结构中，完善数据库表。最终结果可以导出launcher.db查看：

数据库中有数据了，接下来就是要将数据库中的内容绑定到桌面上

bindworkspace()

首先获取在loadworkspace()方法中保存在数据结构里的数据，然后对这些数据进行过滤，区分出当前屏幕的数据和其他屏幕的数据。之后就正式进入数据绑定流程。

  • startBinding()
  • bindworkspaceScreens();
  • bindworkspaceItems()
  • finishBindingItems()

这些方法都是回调Launcher中对应方法完成的先看startBinding:

就是一些准备绑定前的准备工作，接着看bindworkspaceScreens，该方法会回调Launcher的bindScreens()，既然要把应用绑定到桌面上，起码要先把桌面确定下来吧：

这里的bindAddScreens()方法就是创建屏幕:

屏幕确定下来，可以开始绑定了。

bindworkspaceItems()

这里的绑定也没那么纯粹，前面说了会将要绑定的内容分为当前屏幕以及其他屏幕，为了创造好的用户体验，先绑定当前屏幕，再绑定其他屏幕。根据数据的不同，分别回调到Launcehr中的binditem()，bindfolder，bindwidget.就已item为例分析，其他类似：

finishBinding()

各种类型的控件都绑定完成，进入到收尾工作，主要是发送广播
